# Checks Module

## 模块定位
`checks/` 用于在方案设计进入 Framework 和 Output 之前，
对输入内容进行系统化检查。

它不负责生成方案，
而是先判断当前输入是否具备进入下一阶段的条件。

## 核心任务
本模块主要完成五件事：

1. 检查输入是否有缺项
2. 检查信息之间是否有冲突
3. 检查逻辑链条是否一致
4. 检查责任是否归位
5. 判断是否可以进入下一阶段

## 适用场景
本模块适用于：

- 需求整理完成后的进入框架前检查
- 多方输入合并后的冲突检查
- AI 生成方案前的输入质量校验
- 方案初稿生成前的结构预检
- 项目启动前的责任与边界确认
- 复盘时定位失效点来源

## 检查范围
默认覆盖以下五类检查：

### A. 输入完整性检查
检查关键字段是否齐全，是否具备进入下一步的基础条件。

### B. 信息冲突检查
检查不同输入之间是否存在事实、目标、时间、范围、优先级等冲突。

### C. 逻辑一致性检查
检查目标、路径、资源、约束、输出之间是否能形成闭环。

### D. 责任归位检查
检查谁决策、谁执行、谁配合、谁验收是否明确。

### E. 风险暴露检查
检查当前输入中已显性的风险，以及可能导致后续失效的高概率问题。

### F. 条件放行检查
除了检查“字段是否齐”，还要检查“条件是否足以放行”。

至少检查以下问题：

- 是否存在高优先级 `pending` 条件
- 是否存在会影响方案方向的未确认冲突
- 是否存在被默认忽略的关键约束
- 是否存在没有 owner 的条件
- 是否存在 condition 已变化但未同步到框架层
- 是否存在例外条件但未记录为 exception

输出时统一分三类：

- **可放行**
- **带条件放行**
- **不可放行**

---

### G. 人工关口决策
当系统无法自动判断是否进入下一阶段时，
必须生成正式的人工关口记录，而不是只写一句“等待人工确认”。

标准格式：

- **gate_id**：
- **gate_name**：Input Gate / Check Gate / Framework Gate / Output Gate
- **decision**：pass / revise / block / override
- **decision_reason**：
- **confirmed_conditions**：
- **changed_conditions**：
- **unresolved_conditions**：
- **next_module**：
- **reviewer**：
- **review_time**：

说明：
- `pass`：通过，进入下一层
- `revise`：退回修改后重提
- `block`：阻断，不得继续
- `override`：带明确责任的人工覆盖放行

---

### H. 放行条件
列出本轮检查后，可以进入下一阶段的前提条件。

示例格式：

- 条件 1：试点范围已确认
- 条件 2：责任方已明确
- 条件 3：时间口径已统一

---

### I. 阻塞条件
列出导致当前不能进入下一阶段的条件。

示例格式：

- 阻塞条件 1：验收标准缺失
- 阻塞条件 2：预算上限未确认
- 阻塞条件 3：目标对象存在两个版本且未统一

## 本模块输出
每次检查后，统一输出五部分：

- A. Checklist 对照结果
- B. 缺项清单
- C. 风险提示
- D. 需人工确认的冲突点
- E. 逻辑一致性与责任归位判断
