# AI Workflow

> 展示仓试运行版：保留母仓核心架构，迁移核心工作流，供客户先行测试。

本仓库的 AI 工作流采用：

**AI 预处理 + 条件录入 + 人工关口 + AI 继续推进 + 回流沉淀**

的协作方式运行。

它不是“AI 自动生成方案”的直跑流程，  
而是一条把方案设计拆成可输入、可检查、可放行、可回退、可沉淀的工程化主链路。

---

## 工作流主链

### 1. `inputs/`
负责承接原始输入，并先完成输入整理，而不是直接生成方案。

本层主要任务：

- 提取需求重点
- 识别缺失字段
- 标记信息冲突
- 生成补充提问清单
- 记录条件输入数据
- 区分已确认条件与待确认条件

本层输出应至少包括：

- A. 需求重点
- B. 缺失字段
- C. 信息冲突
- D. 补充提问清单
- E. 条件输入数据
- F. 已确认条件
- G. 待确认条件

---

### 2. `checks/`
负责对输入进行检查，不让混乱输入直接进入结构层。

本层主要任务：

- 对照 checklist 检查缺项
- 输出风险提示
- 检查逻辑一致性与责任归位
- 判断当前条件是否足以进入下一阶段
- 生成放行条件与阻塞条件
- 必要时触发人工关口决策

本层输出应至少包括：

- Checklist 对照结果
- 缺项清单
- 风险提示
- 需人工确认的冲突点
- 逻辑一致性与责任归位判断
- 放行 / 阻塞建议

---

### 3. 人工关口：Input / Check Gate
这一关口不只是“等待人工补充”。

人工必须至少留下以下两类记录之一：

#### A. 条件记录
- 新增条件
- 修改条件
- 驳回条件
- 确认条件
- 覆盖默认条件

#### B. 关口决策
- pass
- revise
- block
- override

建议最少记录字段：

- `gate_id`
- `gate_stage`
- `decision`
- `reason`
- `changed_conditions`
- `owner`
- `timestamp`
- `next_action`

如果人工节点没有留下结构化记录，  
则该轮人工介入不能视为已沉淀。

---

### 4. `frameworks/`
负责消费上游已确认条件，完成结构拆解与框架生成。

本层主要任务：

- 读取上游已确认条件
- 标记待确认条件对框架的影响
- 按八个母维度完成拆解
- 提炼方案主线
- 形成结构框架
- 记录本层新增条件
- 输出框架放行结论

本层输出应至少包括：

- 母维度拆解结果
- 结构框架主线
- 权责接口
- 风险节点
- 时间安排
- 本层新增条件
- 框架放行结论

---

### 5. 人工关口：Framework Gate
这一关口用于判断框架是否成立，而不是默认继续生成。

人工应重点判断：

- 当前框架是否成立
- 哪些部分只能带条件继续生成
- 哪些部分必须先回退修正
- 是否应回退到 `checks/`
- 是否允许进入 `outputs/`

建议最少记录字段：

- `gate_id`
- `gate_stage`
- `decision`
- `reason`
- `affected_modules`
- `rollback_target`
- `owner`
- `timestamp`

---

### 6. `outputs/`
负责将框架转化为正式方案草案。

本层主要任务：

- 生成正式方案草案
- 对待确认条件采用条件化写法
- 标记正式方案中依赖条件的部分
- 保持输出结构完整、责任清晰、风险可读
- 为后续案例对照与评审提供标准输出对象

本层输出应至少覆盖：

- 项目背景
- 目标说明
- 核心路径
- 权责接口
- 执行步骤
- 风险预警
- 结果指标
- 复盘方式

---

### 7. 对照 `cases/`
正式输出后，不直接把结果当结论，而要先与案例层对照。

本层主要任务：

- 对照历史成功案例与失败案例
- 判断当前方案与既有案例的相似点与偏差点
- 判断新增条件是否属于既有问题类型
- 判断是否需要新增案例类别
- 为 review 归因提供参考样本

---

### 8. 人工关口：Attribution / Case Gate
这一关口用于判断案例归因与案例归类是否成立。

人工应重点判断：

- 案例归因是否成立
- 主归因 / 次归因是否清楚
- 案例标签是否需要调整
- 是否应新增案例类别
- 当前内容是否可以进入案例沉淀

建议最少记录字段：

- `gate_id`
- `gate_stage`
- `decision`
- `case_type`
- `main_cause`
- `secondary_cause`
- `reason`
- `owner`
- `timestamp`

---

### 9. `reviews/`
负责在问题出现后完成定位、归因、回退判断与修正建议。

本层主要任务：

- 定位问题表现
- 完成主归因 / 次归因判断
- 输出修正建议
- 绑定相关 `condition_id` 与 `gate_id`
- 判断应回退到哪一层修正
- 判断是否需要更新模板、检查点、工作流或提示词

本层输出应至少包括：

- 问题表现
- 影响范围
- 主归因 / 次归因
- 回退层级
- 修正建议
- 是否需要补规则 / 补模板 / 补检查点

---

### 10. `prompts/`
负责根据问题与修正结果，迭代 AI 协作接口。

本层主要任务：

- 根据本轮问题与修正结果迭代 AI 协作提示词
- 补充重点词标注与新提示结构
- 同步新增条件类型与人工关口口径
- 明确 prompt 服务哪一层，而不是越权替代人工判断

本层边界必须明确：

- prompt 负责推进
- prompt 负责整理
- prompt 负责暴露问题
- prompt 不负责最终拍板
- prompt 不负责跳过人工节点

---

### 11. 回流沉淀：`cases/`
最终沉淀不是简单存档，而是把本轮结果转化为系统资产。

本层主要任务：

- 将本轮结果编入成功案例或失败案例
- 将问题样本回流为模板、检查点、风险清单、归因样本
- 必要时更新 `.github/` 协作骨架
- 必要时更新 issue form / PR template / workflow 守护逻辑

可回流资产包括但不限于：

- 新模板
- 新检查点
- 新时间数据
- 新责任归因样本
- 新风险清单
- 新案例标签
- 新 prompt 接口
- 新 GitHub 协作规则

---

## 主链路顺序

当前展示仓试运行版以以下顺序为准：

```text
inputs/
→ checks/
→ 人工补充与修改材料
→ frameworks/
→ outputs/
→ 对照 cases/
→ 人工判断并修改案例归类
→ reviews/
→ prompts/
→ cases/
```
