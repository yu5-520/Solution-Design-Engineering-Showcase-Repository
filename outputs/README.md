# Outputs Module

## 模块定位
`outputs/` 用于把已经完成输入整理、检查校验、框架拆解后的内容，
正式转化为可交付的方案草案。

这一层不再只是做结构判断，
而是要求形成一版完整、可读、可讨论、可执行、可复盘的正式输出。

输出目标不是把内容写长，
而是保证方案草案至少覆盖关键决策所需的信息。

---

## 输出层新增定位
`outputs.md` 不只是把框架层内容写成长文，
而是要把“已确认条件”“待确认条件”“框架层新增条件”一起转成正式方案表述。

这一层的核心任务有三件事：

1. 把框架骨架转成完整方案草案
2. 把条件转成正式文本中的前提、限制、例外、依赖项
3. 明确标出哪些内容已可执行，哪些内容只能在条件成立后执行

输出层不是替上游补判断，
而是忠实承接上游判断，并把判断写成可讨论、可执行、可追责的方案文本。

## 核心要求
方案草案必须至少覆盖以下内容：

1. 项目背景
2. 问题定义
3. 目标设定
4. 核心策略
5. 执行步骤
6. 部门协作方式
7. 时间拆分
8. 风险预警
9. 资源需求
10. 结果指标
11. 复盘方式

只要缺少其中关键模块，
方案就容易出现“能讲故事，但不能执行”或“能执行，但不能评估”的问题。

## 适用场景
本模块适用于：

- 商业方案初稿输出
- 项目执行方案输出
- 运营/营销方案输出
- 跨部门协作方案输出
- 顾问式提案草案输出
- AI辅助生成正式方案文本

## 标准处理流程
### 1. 先对齐框架层结果
输出层不能脱离 `frameworks/` 单独发挥。
正式草案必须承接以下结构结果：

- 方案目标
- 主要路径
- 关键步骤
- 权责接口
- 风险节点
- 时间安排
- 预期结果

输出层的任务，是把这些骨架转成完整方案文本。

---

### 1.5 进入输出层前必须读取的内容
在生成正式方案前，必须先读取以下三类上游信息：

#### A. 框架层结果
- 方案目标
- 主要路径
- 关键步骤
- 权责接口
- 风险节点
- 时间安排
- 预期结果

#### B. 已确认条件
这些内容可以直接写入正式方案，作为成立基础。

示例格式：
- **condition_id**：
- **condition_type**：constraint / decision / exception
- **content**：
- **status**：confirmed
- **impact_module**：output

#### C. 待确认条件
这些内容不能被写成既定事实，
只能写成“条件成立时”的方案表述。

示例格式：
- **condition_id**：
- **condition_type**：assumption / conflict
- **content**：
- **status**：pending
- **risk_note**：
- **should_mark_in_draft**：yes / no

#### D. 框架层新增条件
如果某条路径在 `frameworks.md` 中新生成了前提条件，
输出层必须同步读取，不得遗漏。

示例格式：
- **condition_id**：
- **stage**：framework
- **content**：
- **status**：pending / confirmed
- **next_action**：

### 2. 按固定模块展开
所有方案草案，统一按标准模块展开，
避免每次输出口径不一致，导致后续讨论失焦。

### 3. 保持“可讨论 + 可执行 + 可复盘”
每一部分内容都要至少满足三种要求之一：

- 能帮助决策
- 能指导执行
- 能支持复盘

纯描述性、装饰性、无判断价值的内容应压缩。

---

### 3.5 条件化写作规则
正式方案中，不能把所有内容都写成已确定结论。

应按条件状态使用不同写法：

#### 1. 已确认条件
可直接写成确定表述。

示例：
- 本次试点范围为华东区 20 家门店。
- 项目验收由运营负责人和区域经理共同完成。

#### 2. 待确认条件
必须写成条件化表述。

示例：
- 若 5 月 20 日前完成数据权限开通，则第一阶段按原计划启动。
- 若预算上限维持在 30 万以内，则本方案优先采用轻量试点路径。
- 当前时间节点存在两个版本，正式排期需待人工统一后锁定。

#### 3. 例外条件
必须单独标出，不得藏在正文里。

示例：
- 除直营门店外，加盟门店暂不纳入第一阶段试点。
- 如地方团队已有独立流程，则允许按例外流程接入。

#### 4. 覆盖性人工决策
如果某项内容是人工 override 放行，
必须显式标注“该部分为人工确认口径”。

示例：
- 经人工确认，本轮先按总部统一验收口径推进，后续如组织口径调整，再回流修正。

## 各模块的条件写入要求
正式方案草案的每个模块，除了写内容本身，还必须写明它依赖什么条件。

### 1. 项目背景
必须写明：
- 现实背景是否已确认
- 当前业务环境是否存在冲突口径
- 为什么现在推进，而不是之后推进

### 2. 问题定义
必须写明：
- 当前问题是否已被统一定义
- 问题边界是否已确认
- 是否存在不同角色对问题的不同解释

### 3. 目标设定
必须写明：
- 哪些目标已确认
- 哪些目标仍是阶段性假设
- 目标之间是否存在优先级冲突

### 4. 核心策略
必须写明：
- 策略成立依赖哪些前提条件
- 哪些条件未成立时需切换为备选路径
- 当前策略为何优于其他路径

### 5. 执行步骤
必须写明：
- 每一步是否存在前置条件
- 哪些步骤可直接执行
- 哪些步骤需等待人工确认后执行

### 6. 部门协作方式
必须写明：
- 权责接口是否已确认
- 哪些部门职责已锁定
- 哪些协作接口仍需补充确认

### 7. 时间拆分
必须写明：
- 时间表是否建立在单一口径上
- 哪些里程碑受待确认条件影响
- 是否存在条件成立后才锁定的节点

### 8. 风险预警
必须写明：
- 哪些风险来自资源、时间、组织、权限等条件不稳定
- 哪些风险是上游已暴露但未解决的
- 哪些风险需要单独进入人工关口

### 9. 资源需求
必须写明：
- 已确认资源
- 缺失资源
- 资源上限或例外条件

### 10. 结果指标
必须写明：
- 指标是否与目标一致
- 指标是否依赖外部条件成立
- 指标是否具备验收口径

### 11. 复盘方式
必须写明：
- 复盘基准是什么
- 哪些问题要回流到条件层
- 哪些问题要回流到模板与检查点

- `delivery-format.md`：交付格式与版本规范
- `examples/`：真实输出样例

## 输出层结尾必须附带的摘要
每一版正式方案草案末尾，都要附带以下摘要：

### A. 本版方案状态
- **draft_status**：ready / conditional_ready / blocked

### B. 已确认条件
- 条件 1：
- 条件 2：
- 条件 3：

### C. 待确认条件
- 条件 1：
- 条件 2：
- 条件 3：

### D. 本版草案中受条件影响的部分
- 模块：
- 影响说明：
- 当前写法：确定写法 / 条件写法 / 暂缓写法

### E. 是否需要进入人工关口
- **should_trigger_human_gate**：yes / no
- **gate_type**：Output Gate / Attribution Gate / Review Gate
- **reason**：

### F. 下一步动作
- 直接进入 review
- 退回 framework
- 退回 checks
- 转人工确认
