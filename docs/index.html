示例 01｜创业团队

示例 02｜顾问团队

顾问交付流程验证

怎么测：拿一类常见客户需求做试点，验证模板是否足够复用、主链是否足够稳定。

测完得到什么：可继承的方案结构、可复用的评审节点，以及第一轮可沉淀的交付底色。

示例 03｜企业内部试点

单团队方案流程试点

怎么测：先选一个团队、一个真实需求、一个最小场景，不追求全量智能表现，只验证跑通。

测完得到什么：清晰的接入边界、工作流是否可承载、以及是否值得进入更深层接入的判断依据。

FAQ

客户最容易先问的四个问题

展示仓和母仓有什么区别？

展示仓负责展示、试运行、测试与接入沟通；母仓负责承载 Python 智能核心、深层自动化与后续演进能力。展示仓是试运行载体，不是母仓的完整镜像。

为什么展示仓能运行，但不开放 Python 核心？

因为当前采用的是分层承载策略。展示仓的目标是让客户验证流程能不能跑通，而不是直接开放母仓核心能力。这样既能测试，也能保护母仓的核心智能实现。

试运行后我能拿到什么？

通常能拿到一版结构化输入整理结果、缺失字段清单、风险点、框架骨架、以及是否值得继续接入的判断依据。试运行的重点是验证承载，不是一次性给出完整智能版本。

怎么进入正式合作？

先提交测试需求或先跑一次试运行工作流，验证场景是否适配。试运行通过后，再根据项目情况进入顾问式合作、项目式合作或更深层的企业内部接入。

母仓保护原则

分层承载，而不是一次性全开放

当前体系采用“母仓承载核心、展示仓承载试运行”的方式。 母仓继续保留 Python 智能核心、深层自动化与后续演进能力； 展示仓则负责展示、试运行、测试与接入沟通。
